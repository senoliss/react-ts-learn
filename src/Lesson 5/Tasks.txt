✅ LESSON 5 — Lifting State Up (Concept Explanation Only)
No coding yet — only understanding.

✅ 1. Why “lifting state up” matters
Right now each component manages its own data:

    HealthBar → has its own health state
    ManaBar → has its own mana state
    XPBar → has its own XP state
    Score → has its own score state

This works fine individually, but not for a real game.
✅ Problem:
If each component manages its own data, then:

    They cannot interact.
    Game logic cannot coordinate them.
    You cannot save game state.
    You cannot reset everything at once.
    You cannot share values across UI.

Example:
Your XPBar cannot tell the game to “level up” when XP reaches 100 — because the game doesn’t know XP changed.
Your HealthBar cannot cause a “defeat screen” when health reaches 0.
Your Score component cannot update UI outside itself.
To build a real game UI:

✅ All game state must live in one place
➡ That place becomes the “game brain”
➡ Components only display and send events upward
This is exactly what “lifting state up” means.

✅ 2. What “Lifting State Up” means
❌ Bad approach:
Each component holds its own state:
    HealthBar:   health = 100
    ManaBar:     mana = 100
    XPBar:       xp = 0
    Score:       score = 0

These states do not know about each other.
✅ Good approach:
Create ONE parent component that stores all game state:
GamePanel.tsx
GamePanel:
    health
    mana
    xp
    score

Then GamePanel passes values + callbacks to children:
<GamePanel>
   ├── HealthBar (receives health, onTakeDamage, onHeal)
   ├── ManaBar   (receives mana, onCastSpell, onRegen)
   ├── XPBar     (receives xp, onGainXP)
   ├── Score     (receives score, onAddScore)
</GamePanel>

This is the core idea.

✅ 3. What the parent component must do

The parent must:
✅ Hold the main game state

    const [health, setHealth] = useState<number>(100);
    const [mana, setMana] = useState<number>(100);
    const [xp, setXp] = useState<number>(0);
    const [score, setScore] = useState<number>(0);

✅ Define logic functions

    function takeDamage(amount: number) { ... }
    function heal(amount: number) { ... }
    function castSpell(amount: number) { ... }
    function gainXP(amount: number) { ... }
    function addScore(amount: number) { ... }

✅ Pass state + functions down via props
Example:

<HealthBar 
   health={health} 
   maxHealth={100}
   onDamage={() => takeDamage(10)}
   onHeal={() => heal(10)}
/>

Components no longer modify state themselves — they only tell the parent what happened.

✅ 4. Child Components Become “Dumb” Components

After lifting state up, your components:
    ✅ display data
    ✅ trigger callbacks
    ❌ no longer contain game logic
    ❌ no longer hold health/mana/xp score state

Example:
Old ManaBar:

    internal useState
    internal castSpell
    internal regenerateMana

New ManaBar:

    receives mana and maxMana as props
    receives onCastSpell callback
    receives onRegenMana callback
    no useState inside

This makes the components reusable AND consistent.

✅ 5. Why this matters for your mini‑game

After lifting state up, we can do:
    ✅ Level-up when XP reaches 100
    ✅ Disable spell cast button when mana < cost
    ✅ Show warning when health < 30%
    ✅ Reset entire game with one button
    ✅ Save the entire game state later
    ✅ Use Context API to share state across pages
    ✅ Make a custom hook like useGameState()
Everything becomes possible once state is centralized.

✅ 6. Visual Summary

Before (Bad):

After (Good):
